# 【初心者向け】ターミナルからWebアプリをデプロイするまでのGit完全ガイド

## 📖 はじめに

本ドキュメントは、Webアプリケーション開発における **「Gitを用いたデプロイフロー」** を、初学者向けに体系化した技術ガイドです。
私自身が学習過程でつまずいたポイントや、理解を助けた「メンタルモデル（概念のイメージ）」を中心に解説しています。

### このドキュメントの目的
1.  **知識の定着**: コマンドの暗記ではなく、背後にある仕組みを理解する。
2.  **実務への適応**: チーム開発やデプロイ自動化（CI/CD）の基礎となる操作を習得する。
3.  **ナレッジシェア**: 過去の自分と同じ悩みを持つ初学者へ、再現性のある手順を提供する。

---

## 🛠 前提となる概念：Gitの3つのエリア

作業を始める前に、Gitがファイルをどう管理しているか、その「場所」をイメージすることが重要です。

| エリア名 | 役割 | 日常生活での比喩 |
| :--- | :--- | :--- |
| **ワークツリー (Work Tree)** | 現在作業しているフォルダ | **机の上**（作業中の書類） |
| **ステージングエリア (Index)** | コミットするファイルを置く場所 | **配送用の箱**（発送する商品を選んで入れる） |
| **リポジトリ (Repository)** | 変更履歴が記録される場所 | **配送センターの記録台帳**（発送済み履歴） |

## 🚀 ハンズオン：ゼロからGitHubへデプロイする5つのステップ

ここからは実際のターミナル操作を通じて、ローカル環境（自分のPC）にあるプロジェクトをGitHub（リモート環境）へアップロードする手順を解説します。

### 事前準備
- ターミナル（Mac）または Git Bash / PowerShell（Windows）を開いてください。
- 対象となるプロジェクトのディレクトリに移動しておきます。
  ```bash
  cd path/to/your/project
  ```

### Step 1: プロジェクトの初期化（git init）

普通のフォルダを「Git管理下のフォルダ」へ変換します。

```bash
git init
```

- **何が起きるか**: フォルダの中に不可視の `.git` ディレクトリが生成されます。
- **技術解説**: これ以降、このフォルダ内のファイルの変更（追加・削除・編集）はすべてGitによって監視（トラッキング）されます。
- **注意点**: 1つのプロジェクトにつき、最初に1回だけ実行します。

### Step 2: ステージング（git add）

変更されたファイルを「次のコミット（記録）」に含めるために選びます。

```bash
git add .
```

- **コマンドの意味**: `.` は「カレントディレクトリの全てのファイル」を指します。
- **Web制作との対比**: WordPressで記事を書いた後、「公開」ボタンを押す前に**「チェックボックスを入れて公開対象を選ぶ」**作業に相当します。
- **なぜ必要か**: 作業中のファイル全てを保存したいとは限らないため、保存するファイルを明示的に選ぶ工程が存在します。

### Step 3: コミット（git commit）

ステージングエリアにあるファイルを、ひとつの「歴史の記録」として確定させます。

```bash
git commit -m "Initial commit: プロジェクトの基本構成を作成"
```

- **コマンドの意味**: `-m` オプションを使って、変更内容の説明（コミットメッセージ）を付与します。
- **重要な挙動**: これを実行して初めて、変更内容が `.git` ディレクトリ内のデータベースに永続的に保存されます。
- **実務のポイント**: コミットメッセージは「何をしたか」が第三者に伝わるように具体的に記述します（例: "バグ修正" ではなく "ヘッダーのロゴ表示崩れを修正"）。

### Step 4: リモートリポジトリの登録（git remote）

自分のPC（ローカル）にあるGitに、アップロード先（GitHub）の住所を教えます。
*※事前にGitHub上で `New Repository` を作成し、URLを取得しておく必要があります。*

```bash
git remote add origin https://github.com/YourUsername/YourRepoName.git
```

- **解説**:
    - `remote add`: 遠隔地の接続先を追加する。
    - `origin`: 接続先の「ニックネーム」。慣習的にメインの接続先は `origin` と呼びます。
- **確認方法**: `git remote -v` と入力すると、登録されたURLが確認できます。

### Step 5: プッシュとデプロイ（git push）

ローカルのコミット履歴をGitHubへアップロードします。
現代のWeb開発では、この操作がデプロイ（公開）のトリガーとなることが一般的です。

```bash
git push -u origin main
```

- **コマンド分解**:
    - `push`: 送信する。
    - `-u`: "Upstream" の略。次回以降 `git push` だけで送信できるように、紐付けを記憶させるオプション（初回のみ必要）。
    - `origin`: Step 4で登録した送信先。
    - `main`: 送信するブランチ（履歴のライン）の名前。
- **成功の証**: ターミナルに `Branch 'main' set up to track remote branch...` と表示されれば完了です。

## ⚠️ トラブルシューティング（よくあるエラーと対処）

初学者が遭遇しやすいエラーとその解決策をまとめました。

### Case 1: "fatal: not a git repository..."
- **原因**: `git init` を実行していないフォルダでGitコマンドを使おうとしています。
- **対処**: 正しいフォルダに `cd` しているか確認し、必要であれば `git init` を実行してください。

### Case 2: "Author identity unknown"
- **原因**: Gitに「誰が変更したか」というユーザー情報が登録されていません。
- **対処**: 以下のコマンドで設定を行います。
  ```bash
  git config --global user.name "あなたの名前"
  git config --global user.email "あなたのメールアドレス"
  ```

### Case 3: "error: failed to push some refs to..."
- **原因**: GitHub上の履歴と、PC上の履歴が整合していない（例：GitHub上で直接ファイルを編集してしまった）。
- **対処**: まずGitHub側の変更をPCに取り込む必要があります。
  ```bash
  git pull origin main
  ```
  その後、再度pushを試みます。

## 🎓 まとめ

Gitによるデプロイフローは、以下のサイクルの繰り返しです。

1.  **編集する**（作業）
2.  **`git add`**（選択）
3.  **`git commit`**（記録）
4.  **`git push`**（公開）

このフローを理解することは、単なるツール操作の習得にとどまらず、**「変更を安全に管理し、チームと共有する」**というエンジニアとしての必須スキルの一部です。
